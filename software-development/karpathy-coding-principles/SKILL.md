# Karpathy 编码四原则

源自 Andrej Karpathy 的 LLM 编码陷阱观察，65k+ ⭐ 验证。

## 何时使用

当用户提出**开发类任务**时，自动触发以下检查：

- 方案设计 → 先呈现假设和权衡，再执行
- 改写代码 → 确认改动的边界，只动该动的
- 多步任务 → 定义可验证的成功标准，再推进
- 代码评审 → 优先指出过度设计，不只是风格

## 四原则

### 1. Think Before Coding（编码前先想）

**核心：不要假设，不要隐藏困惑，主动呈现权衡。**

触发场景：
- 不确定需求有几种解释 → 先列出来，让用户选
- 发现更简单的方案 → 直接提出来，不用等问
- 对某个点感到困惑 → 停下来问清楚再动
- 要做技术决策 → 列出选项和 tradeoffs

执行：
```
假设：
- A: ...
- B: ...
推荐：A（理由）
是否同意？
```

### 2. Simplicity First（简单优先）

**核心：最小代码解决问题，不做任何没被要求的事。**

禁止清单：
- ✗ 添加没被要求的功能
- ✗ 为单次使用创建抽象层
- ✗ 预留"灵活性"或"可配置性"
- ✗ 处理不可能发生的错误场景
- ✗ 200 行可以 50 行解决却不简化

自检问题：**"一个 senior 工程师会说这过度复杂吗？"** 如果是，简化。

### 3. Surgical Changes（手术式修改）

**核心：只碰必须碰的，清理自己造成的孤儿代码。**

编辑现有代码时：
- ✗ 不"优化"相邻代码/注释/格式
- ✗ 不重构没坏的东西
- ✗ 不按自己的风格改而无视现有风格
- ✓ 如果发现无关的死代码 → 提及但不动它

自己造成孤儿时：
- ✓ 删除因本次改动而废弃的 import/变量/函数
- ✗ 不删除改动前已存在的死代码（除非被要求）

**判断标准：每一行改动都要能追溯到用户的原始请求。**

### 4. Goal-Driven Execution（目标驱动执行）

**核心：把任务翻译成可验证的成功标准，然后循环验证。**

转化模式：

| 用户说 | 转化为 |
|--------|--------|
| "加验证" | "先写测试覆盖非法输入，再让测试通过" |
| "修 bug" | "先写测试复现 bug，再修复" |
| "重构 X" | "确保重构前后测试都通过" |

多步任务模板：
```
1. [步骤] → 验证：[检查点]
2. [步骤] → 验证：[检查点]
3. [步骤] → 验证：[检查点]
```

**强成功标准 → 自主循环；弱成功标准（"让它工作"）→ 持续需要澄清。**

## 工作流集成

```
用户开发请求
    ↓
原则 1：编码前
    ↓ 假设清单 + 方案选择
用户确认 / 补充
    ↓
原则 2：设计方案
    ↓ 最小实现路径
开始执行
    ↓
原则 3：手术式修改
    ↓ 每一行追溯到原始请求
    ↓ 清理自己的孤儿代码
    ↓
原则 4：目标驱动
    ↓ 可验证标准 + 循环验证
交付
```

## 判断这些原则是否在起作用

- diff 里没有多余的改动
- 代码第一次就足够简单，不需要返工
- 问题是实现前问出来的，不是出错后补救的
- PR 干净，没有顺路"优化"的无关改动

## 局限

这些原则偏向**谨慎胜于速度**。对于简单任务（明显的 typo、一行修复），运用判断力，不需要完整走流程。
