# Debugging Reasoning Framework

## 四种高级推理框架

### 1. 反事实调试（Counterfactual Debugging）
**核心问题**：不问"为什么失败"，问"什么最小改动能让它成功"

```python
# 操作方法：
# 1. 列出所有"让它成功"的候选改动
# 2. 按改动幅度从小到大排序
# 3. 逐个验证最小必要改动
#
# 优势：快速隔离必要因素，不被冗余信息干扰
# 适用：单点故障、配置错误、类型不匹配
```

### 2. 差异调试（Differential Debugging）
**核心问题**：对比两个应相同但不同的环境

```python
# 操作方法：
# 1. 找到"标准参照"（官方示例/另一个正常实例）
# 2. 对比：我的代码 vs 标准参照
# 3. 差异点 = 故障点
#
# 适用：环境差异、库版本差异、配置差异
# 工具：diff, git diff, pytest --assert
```

### 3. 契约调试（Contract Debugging）
**核心问题**：先验证组件间契约是否被违反

```python
# 操作方法：
# 1. 列出组件间的"契约"：
#    - 输入格式/类型
#    - 输出格式/类型
#    - 副作用（状态修改、I/O）
# 2. 从最外层向内逐层验证契约
# 3. 找到第一个被违反的契约
#
# 适用：模块化系统、API调用、复杂状态机
# 优势：避免在错误层级调试
```

### 4. 时间切片调试（Temporal Slicing）
**核心问题**：故障是时序耦合的，不只是空间耦合

```python
# 操作方法：
# 1. 找到"故障最早出现的时间点"T故障
# 2. 找到"最后正常状态"的时间点T正常
# 3. 二分查找：在T故障和T正常之间，找到故障被引入的瞬间
# 4. 检查该时刻的变更：谁、改了什么、影响了什么
#
# 适用：分布式系统、并发bug、race condition
# 关键信号：T1修改 → T3故障，但T2看起来一切正常
```

## 常见盲点（避免）

| 盲点 | 说明 | 对策 |
|------|------|------|
| **假设生成偏向性** | 倾向于从最近修改处生成假设 | 强制检查"最老代码中最可疑的部分" |
| **验证的确认偏误** | 验证通过≠假设正确 | 主动构造"最能否定假设"的测试用例 |
| **忽略静默失败** | 只追踪报错，不追踪"应该发生但没发生" | 在日志中搜索"should have"和"expected but didn't" |
| **上下文污染** | 用上一次调试的假设套用 | 每个新bug强制"清空假设" |

## 调试 Checklist

- [ ] **反事实**：最小成功改动是什么？
- [ ] **差异**：有没有可对比的正常参照物？
- [ ] **契约**：组件间契约是否被违反？
- [ ] **时序**：T1修改→T3故障的因果链是什么？
- [ ] **清空假设**：这个bug真的是我以为的原因吗？
